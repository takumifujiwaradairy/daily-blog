## I Didn't Understand My Own Product

I work as an engineer at a startup. I've been building a business planning tool that helps users fill out frameworks with AI — but honestly, I didn't really understand what a Business Model Canvas (BMC) was.

A BMC is a framework that maps out a business in 9 blocks: who you serve, what value you deliver, how you reach them, and how you make money. All on one page.

```
┌───────────┬───────────┬───────────┬───────────┬───────────┐
│           │           │           │           │           │
│  Key      │  Key      │           │  Customer │  Customer │
│  Partners │  Activities│  Value    │  Relations│  Segments │
│           │───────────│  Proposition──────────│           │
│           │  Key      │           │           │           │
│           │  Resources│           │  Channels │           │
│           │           │           │           │           │
├───────────┴───────────┴───────────┴───────────┴───────────┤
│  Cost Structure                │  Revenue Streams          │
└────────────────────────────────┴──────────────────────────┘
```

Our product lets users fill in these 9 blocks while sparring with AI. And yet, I didn't understand the meaning or intended workflow of the framework I was building for.

## Right to Left Is the Standard

There's a standard order for filling in a BMC: start from the right (customer) and work left (your business).

1. First: "Who is this for?" (Customer Segments)
2. Then: "What problem do we solve?" (Value Proposition)
3. Then work left from there

The reason is simple: if you start with "what can I build?" instead of "who needs help?", you end up building something nobody wants.

This made me notice something. Our product's input form follows the visual layout order (top-left first), not this right-to-left workflow. There's room for improvement.

## What If We Split the JSON Into 9 Tables?

Currently, our product stores all 9 BMC blocks in a single JSON column:

```json
{
  "keyPartners": { "q1": "...", "q2": "...", "q3": "..." },
  "customerSegments": { "q1": "...", "q2": "...", "q3": "..." },
  ...
}
```

What if we split this into 9 separate tables? After learning the domain, the answer became clear.

### Track How Ideas Evolve

With a history table per block, you can record how a business concept evolves over time:

```
Customer Segments — History
v1 (4/1)  → "Women in their 20s, living alone"
v2 (4/15) → "Dual-income couples in their 30s"
v3 (5/1)  → "Dual-income couples in their 30s, no kids"
```

With JSON, you can only save the entire blob. You can't tell which block changed.

### AI Can Catch Contradictions

If blocks are independent, AI can check consistency between them. For example: if Customer Segments says "B2B enterprise" but Channels says "Instagram ads", the AI can flag it: "Does that match?"

### Dashboards Can Show Growth

```
Customer Segments  ████████░░  v3 (stabilizing)
Value Proposition  ██████░░░░  v2 (still shifting)
Channels           ██░░░░░░░░  v1 (not explored)
Revenue Streams    ░░░░░░░░░░  empty
```

You can see which blocks are mature, which are weak, and when a pivot happens — what changed together and what didn't.

## You Can't Design a Database Without Domain Knowledge

This was the biggest takeaway.

I had been doing DB design without understanding how BMC works. Each of the 9 blocks has a specific meaning. Users fill them in from right to left. The content changes over time as the business idea matures. Without knowing this "how it's used" context, there's no way to judge whether JSON is enough or whether you need separate tables.

Engineers tend to focus on "how to implement this technically." But before that, you need to understand "how will this data be used." Without that, you can't arrive at the right design.

It sounds obvious. But I felt it firsthand, building my own product.
