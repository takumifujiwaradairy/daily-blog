## 自分のプロダクトを知らなかった

僕はスタートアップでエンジニアをしている。事業構想支援ツールの開発に携わっているのだが、正直に言うと、ビジネスモデルキャンバス（BMC）が何なのか、ちゃんと理解していなかった。

BMCは事業の全体像を9つのブロックで整理するフレームワークだ。「誰に」「何を」「どうやって届けて」「どう稼ぐか」を1枚の図で可視化する。

```
┌──────────┬──────────┬──────────┬──────────┬──────────┐
│          │          │          │          │          │
│ パートナー │ 主要活動  │          │ 顧客との  │ 顧客      │
│ (KP)     │ (KA)     │  価値提案  │ 関係(CR)  │ セグメント │
│          │──────────│  (VP)    │──────────│ (CS)     │
│          │ 主要      │          │          │          │
│          │ リソース   │          │ チャネル   │          │
│          │ (KR)     │          │ (CH)     │          │
├──────────┴──────────┴──────────┴──────────┴──────────┤
│ コスト構造(C$)              │ 収益の流れ(R$)             │
└────────────────────────────┴──────────────────────────┘
```

この9ブロックをAIと壁打ちしながら埋めていくツールを開発している。なのに、このフレームワークの意味も使い方もわかっていなかった。

## 右から埋めるのが定石

BMCには定番の順番がある。右側（顧客）から左側（自社）へ埋める。

1. まず「誰のため？」（顧客セグメント）
2. 次に「何を解決する？」（価値提案）
3. そこから左へ順に埋めていく

理由はシンプルで、顧客不在のまま「自分が作りたいもの」を先に考えると、誰も欲しがらないものができるから。「こんな技術がある→何か作ろう」ではなく、「この人が困っている→だからこれを作る」の順番。

ここで気づいた。うちのプロダクトの入力フォームは、この定石とは違う順番になっている。図のレイアウト順（左上から）でフォームが並んでいる。ここにはプロダクト改善の余地がある。

## JSONを9テーブルに分けたら何ができるか

今のプロダクトは、BMCの全9ブロックを1つのJSONカラムに格納している。こんな形だ。

```json
{
  "keyPartners": { "q1": "...", "q2": "...", "q3": "..." },
  "customerSegments": { "q1": "...", "q2": "...", "q3": "..." },
  ...
}
```

これを9つのテーブルに分けたらどうなるか。ドメイン知識を学んだ後に考えたら、答えが見えた。

### 変遷が追える

ブロックごとにHistoryテーブルを持てば、事業構想の変遷が記録できる。

```
顧客セグメント の変遷
v1 (4/1)  → 「20代女性・一人暮らし」
v2 (4/15) → 「30代共働き夫婦」
v3 (5/1)  → 「30代共働き夫婦・子なし」
```

JSONだと全体を丸ごと保存するしかないから、「どのブロックが変わったか」の差分が取れない。

### AIが矛盾を指摘できる

ブロックが独立していれば、ブロック間の整合性をAIがチェックできる。例えば、顧客セグメントが「法人」なのにチャネルが「Instagram広告」になっていたら、「それ合ってる？」と指摘できる。

### ダッシュボードで成長が見える

```
顧客セグメント  ████████░░  v3 (安定してきた)
価値提案       ██████░░░░  v2 (まだブレてる)
チャネル       ██░░░░░░░░  v1 (未検討)
収益の流れ     ░░░░░░░░░░  未記入
```

どのブロックが成熟していて、どこがまだ弱いのか。ピボットがあった時に何が連動して変わって、何が変わらなかったのか。全部可視化できる。

## ドメイン知識なしにテーブルは設計できない

これが今日一番の学びだった。

BMCの構造も使われ方も知らない状態でDB設計をしていた。9つのブロックにはそれぞれ意味があって、ユーザーは右から左へ埋めていって、時間とともに内容が変わっていく。その「使われ方」を知らなければ、JSONで十分なのかテーブルを分けるべきなのか、判断のしようがない。

エンジニアは「技術的にどう実装するか」に意識が向きがちだ。でもその前に「このデータはどう使われるのか」を理解しないと、適切な設計にはたどり着けない。

当たり前のことだけど、自分のプロダクトで痛感した。
